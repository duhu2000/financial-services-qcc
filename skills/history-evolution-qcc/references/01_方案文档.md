## 一、定位与总体思路

### 1.1 一句话定位

**以企查查 qcc-history MCP 工具链为核心数据源，输出一份面向银行 KYB 开户尽调（主场景）、兼顾投资尽调（次场景）的「企业历史发展沿革」结构化分析报告。**

### 1.2 主场景 / 次场景

| 维度 | 主场景 · 银行 KYB / 开户尽调 | 次场景 · 投资尽调 / 背景调查 |
|---|---|---|
| 典型用户 | 银行对公客户经理、合规岗、反洗钱岗 | FA / PE / 战投的项目经理、DD 团队 |
| 关注重点 | 避险式变更、信用洗白识别、控制权稳定性、UBO 连续性 | 融资节奏与估值路径、投资方画像、退出事件、股改进度 |
| 数据锚定 | 对齐 Advanced Report (China mainland) 的 CHANGE HISTORY 章节结构 | 同一份报告内以「资本沿革」子章节覆盖 |
| 使用频次 | 高频，新客户建档即查 | 中低频，单笔交易查一次 |

**单报告双场景覆盖**：两类受众所需字段 85% 重合，只是关注点不同，不做版本分叉，通过章节设计与"专家点评"文字差异覆盖。

### 1.3 与现有草稿（QCC_History_Skill_Final_Package）的关系

| 对比项 | 现有草稿 V0 | 本方案 V1.0 |
|---|---|---|
| 场景定位 | 泛"金融尽调" | **主 KYB + 兼投资尽调**，明确主次 |
| 风险章节 | 第六章独立章节，铺开铺全 | **纳入但只做时间轴对撞**，不独立铺展，聚焦工商沿革主线 |
| 报告章节 | 7 章（含独立风险画像） | **7 章**（风险画像合并到沿革主线，新增独立"业务版图与关联方网络"章节） |
| 数据锚定 | 无 | 对标 Advanced Report 的 CHANGE HISTORY 字段 |
| 示例数据 | 无完整示例 | 内嵌「企查查科技」完整真实示例 |
| 交付产物 | 方案 + Prompt + 配置 | 方案 + Markdown 模板 + Word 模板 + PDF |
| 落地校验 | 无 Checklist | 含完整落地 Checklist |

## 二、目标用户与典型问题

### 2.1 用户画像

1. **银行对公客户经理**
   - 问："这家公司开户，历史上干净吗？有没有改过名、迁过址、换过法代？"
   - 问："股东里现在的大股东是什么时候进来的？以前是谁？"
   - 问："有没有那种'风波前夕突然换法代'的苗头？"

2. **银行合规 / 反洗钱岗**
   - 问："UBO 连续性是否能回溯到 2014 国家信用信息公示系统上线？"
   - 问："是否有'信用修复型'操作（曾失信/被执行，现已清除）？"
   - 问："注册资本异常跃升或腾挪？"

3. **投资尽调 DD**
   - 问："融资节奏和里程碑？每轮主要投资方是谁？"
   - 问："退出事件？谁退出、何时退出、被谁接手？"
   - 问："是否已完成股改？股改时间点与 IPO 进度？"

### 2.2 典型问法（用户调用 SKILL 时的自然语言示例）

- 「帮我查一下企查查科技股份有限公司的历史沿革」
- 「生成 XX 有限公司的发展历程报告，重点看控制权变迁」
- 「这家公司要来我行开户，先出一份历史沿革 KYB 报告」
- 「融资尽调 · XX 公司 · 重点看注册资本演变和历次股东变更」

## 三、数据来源与 MCP 工具编排

### 3.1 涉及 3 个 MCP Server

| Server | 用途 | 工具数 |
|---|---|---|
| `qcc-company` | 当前工商基础信息 + 变更记录 | 2 个（本 SKILL 用到） |
| `qcc-history` | 历史存档（工商 + 司法风险 + 经营信息） | 34 个（本 SKILL 用到 11 个） |
| `qcc-executive` | 董监高画像（可选，按需深挖关键人物） | 1 个（按需） |

### 3.2 工具调用清单

| 6 | `get_historical_executives` | qcc-history | 历史董监高任免 |
| 7 | `get_historical_external_investments` | qcc-history | 历史对外投资（已退出的被投企业） |
| 8 | `get_historical_branches` | qcc-history | 历史分支机构（已注销分支） |

| 序号 | 工具 | Server | 触发条件 |

## 四、输入参数设计

### 4.1 用户输入（SKILL 入口）

SKILL 接受 2 种输入形式：

**形式 A · 自然语言**（主）
```
帮我查一下 [企业名称] 的历史沿革，生成 KYB 报告
```
→ LLM 从自然语言中提取 `company_name`。

**形式 B · 结构化参数**（备）
```yaml
company_name: 企查查科技股份有限公司   # 必填
scenario: kyb                          # 可选，默认 kyb；可选值 kyb / investment
time_range: full                       # 可选，默认 full；可选值 full / last_5y / from_<YYYY>
deep_dive_person: true                 # 可选，默认 false；是否触发人员深挖
output_format: docx                    # 可选，默认 docx；可选值 docx / md / both
```

### 4.2 自动推断规则

- `scenario`：若用户问题含"开户/进件/客户经理/KYB/反洗钱/EDD"→ `kyb`；含"投资/融资/估值/FA/交易/退出"→ `investment`；否则 `kyb`（主场景默认）。
- `time_range`：默认 `full`（从企业成立起全量）。用户明说"最近 3 年/最近 5 年"时裁剪。
- `deep_dive_person`：默认关闭，仅当 LLM 识别出"单一自然人在 2 个以上关键节点同时变更"时自动开启。

## 五、报告章节结构（7 章）

### 5.1 章节总览

| 章节 | 标题 | 核心字段 | 对撞点 |
|---|---|---|---|
| 一 | 企业基本概况 | 当前名称、统一社会信用代码、成立日期、当前注册资本、经营状态、当前法代 | — |
| 二 | 名称与品牌演变 | 历史名称（按时间段） | — |
| 三 | 资本实力与融资历程 | 注册资本演变、股东进出、主要融资事件 | 与失信/被执行对撞 |
| 四 | 治理结构与控制权变迁 | 法代变更、董监高进出、大股东更替 | 与行政处罚/诉讼对撞 |
| 五 | 业务发展与经营演变 | 经营范围变更、地址变更 | 与经营异常对撞 |
| **六** | **业务版图与关联方网络**（**新增独立章节**） | **历史对外投资、历史分支机构、历史网站备案** | **与关联方风险对撞** |
| 七 | 总结与尽调建议 | 整体评价（标签）、风险提示、尽调方向 | 综合结论 + 评级标签 |

> **与 V0.9 的调整**：原第五章"业务发展与版图演变"拆为两章——**§ 五"业务发展"**（经营范围 + 地址，内聚"企业主体做什么事、在哪里做"）与 **§ 六"业务版图"**（对外投资 + 分支 + 网站备案，内聚"企业主体延伸出了多少触角"）。业务版图独立成章是为了给关联方网络 / 穿透链条留足表达空间，尤其支持 KYB UBO 穿透与投资尽调的关联方风险审查。
>
> **合规历史与风险画像**不再作为独立章节出现，而是以"⚠️ 关联事件"小字条的形式**嵌入**到第三/四/五/六章的具体变更行下。

### 5.2 每章字段映射

#### 第一章 · 企业基本概况

| 字段 | 来源工具 | 来源字段 |
|---|---|---|
| 当前名称 | get_company_registration_info | company_name |
| 统一社会信用代码 | get_company_registration_info | unified_social_credit_code |
| 成立日期 | get_company_registration_info | established_date |
| 当前注册资本 | get_company_registration_info | registered_capital |
| 当前法代 | get_company_registration_info | legal_representative |
| 经营状态 | get_company_registration_info | status |
| 登记机关 | get_company_registration_info | registration_authority |
| 公司类型 | get_company_registration_info | company_type |

#### 第二章 · 名称与品牌演变

| 字段 | 来源工具 | 来源字段 | 处理规则 |
|---|---|---|---|
| 阶段序号 | — | — | AI 按时间倒序编号：最早阶段 = I，最新 = N |
| 存续期间 | get_historical_registration | name_history[].period | 保留起止，"-至..." 表示成立至某日 |
| 企业名称 | get_historical_registration | name_history[].name | 原样 |
| 关键节点说明 | get_change_records | records[变更项目=名称变更] | LLM 基于上下文生成一句话解读 |

#### 第三章 · 资本实力与融资历程

包含两个子表：

**3.1 注册资本演变表**
| 字段 | 来源 | 处理规则 |
|---|---|---|
| 变更日期 | get_change_records[变更项目=注册资本变更].date | 原样 |
| 变更前 | get_change_records[].before | 提数字 |
| 变更后 | get_change_records[].after | 提数字 |
| 增减幅 | — | 计算 `(after-before)/before × 100%` |
| 关键事件/投资方 | get_historical_shareholders 同期快照 | LLM 关联"同日股东变更"推断融资轮次 |

**3.2 历次股东变更表**
| 字段 | 来源 | 处理规则 |
|---|---|---|
| 变更日期 | get_change_records[变更项目=股东变更].date | 原样 |
| 新进方 | get_change_records[].after 含【新进】的条目 | 提取姓名 + 持股比例 |
| 退出方 | get_change_records[].before 含【退出】的条目 | 提取姓名 + 持股比例 |
| 变动幅度 | 计算 | 新进 / 退出的累计股比 |
| 关联资本事件 | 联动注册资本 / 名称变更 | LLM 标注"同日同步变更" |
| ⚠️ 关联事件 | 与 5 件套风险对撞 | 若 ±3 个月内存在风险事件，在此列标注 |

#### 第四章 · 治理结构与控制权变迁

**4.1 法代变更历程**
| 变更日期 | 变更前 | 变更后 | 同日其他变更 | ⚠️ 关联事件 |
|---|---|---|---|---|
| 2015-05-05 | 马群 | 陈德强 | 股东大洗牌、注册资本 200→224.7 万 | —（本例无触发） |

**4.2 董监高核心变动**
| 变更日期 | 变更项目 | 退出人员 | 新进人员 | 专家点评 |

**4.3 大股东更替（含合伙企业/持股平台穿透、SPV 架构识别）**
以"时间轴 + 持股比例 ≥ 10% 的股东"为主线，呈现控制权重心的迁移。

| 时间锚点 | 主要持股方（≥10%） | 是否合伙/SPV | 穿透后最终受益人 | 备注 | 来源 |
|---|---|---|---|---|---|

**填写规则（与 §6.1 工作准则 10 联动）**：
- 若"主要持股方"名称包含以下关键词：`有限合伙` / `企业管理中心` / `投资中心` / `合伙企业` / `管理合伙` / `员工持股` / `持股平台` / `XX 壹号/贰号/投资基金`，则"是否合伙/SPV"列标注 `Y`，并在下一步：
  - 立即调用 `mcp__qcc-company__get_shareholder_info`（最多 3 层，含 GP/LP 分角色披露）
  - "穿透后最终受益人"列必须写到自然人或国资委层（禁止留"待核查"）
  - 若穿透结果显示受益人未变（如 "Wind 直接→间接"），必须在"备注"列明确标注 `同一受益人换主体 · 非实质退出`
- "是否合伙/SPV = N" 的普通法人/自然人股东，"穿透后最终受益人"列照抄股东名称即可
- "来源"列格式固定为 `[来源: tool · record_id=<ID> · snapshot=YYYY-MM-DD]`（对齐 §7.4 行级溯源）

**6.4 关联方网络专家点评**（模板）

> *关联方网络解读要点*：
> - 被投企业中是否有**当前或历史的失信 / 被执行主体**？
> - 分支机构撤销是否**集中在某个时间窗口**（可能的战略收缩信号）？
> - 历史 ICP 备案主体名称若与现名不一致，**链接到第二章名称演变**是否能自洽？
> - 对外投资是否形成"**产业链上下游投资**"还是"**投机性多元投资**"？

#### 第七章 · 总结与尽调建议

固定 4 小节：

1. **整体发展评价标签（三档固定）** —— 按 §6.1 工作准则第 11 条，从下列 3 个固定标签中选一：
   - `整体评价标签: clean_baseline`（一贯合规）
   - `整体评价标签: recovered`（信用修复型）
   - `整体评价标签: evasive_high_risk`（避险型 / 高风险）

   标签下附 2-4 条关键论据（简述），不得自由措辞替代标签。

2. **关键风险提示（1~3 条）**：对撞出的关联事件在此高亮。每条风险提示**不得留"建议进一步穿透 / 背景未明"等悬念尾巴**，必须已在会话内完成穿透核实（参见 §6.1 工作准则第 9/10 条）。

3. **尽调建议**：KYB 场景侧重"UBO 连续性 / 受益人合并同类项（如合伙企业与其 GP）"；投资场景侧重"股改后 IPO 进度 / 最新轮次投资方背景 / 被投企业网络健康度"。

4. **数据来源清单**：以结构化小表附在报告末尾，列出本次调用的 MCP 工具、调用次数、累计积分消耗，便于运营统计单次 SKILL 成本。

   - 若判定为 investment 场景，第七章侧重"融资节奏、投资方退出、股改进度"。
9. **穿透优先 · 禁止悬念尾巴**：
   - 对报告中任何一个不确定结论，**必须先穿透核实一层 MCP 数据再下结论**，禁止在输出中留下"待核查 / 背景未明 / 建议进一步穿透"等悬念式尾巴。

## 七、数据处理与校验规则

### 7.1 连续性校验

| 校验项 | 规则 | 失败处理 |
|---|---|---|
| 时间轴断层 | 相邻两条记录时间间隔 > 3 年 → 疑似数据缺口 | 在报告中标注"数据提示：2014 年国家信用信息公示系统上线，早期信息可能不完整" |
| 股东进出闭环 | 每一笔新进必须对应一笔等额退出（或新增注册资本） | 失败则在表格备注"数据待核" |
| 注册资本连续 | 注册资本变更的"变更前"必须等于上一条的"变更后" | 失败则标注"⚠️ 数据不连续，建议人工核对" |

### 7.2 去重与归并

- **同日多项变更**：一次工商登记常含多项（如 2023-07-06 企查查科技同日发生 7 项变更），归并为"里程碑事件"整体处理。
- **持股比例微调**：< 0.5% 的股东持股增减视为"程序性调整"，不进入大股东更替表。

### 7.3 脱敏规则

- 本 SKILL 输出的报告**不做脱敏**（KYB / 投资场景均需要真实姓名与企业名）。
- 仅在报告末尾加一段合规声明："本报告数据来自企查查 MCP 实时调用，仅供尽调参考，不构成对任何人明示或暗示的担保或结论。"

### 7.4 行级数据溯源（硬门控）

金融尽调报告要求可审计。本 SKILL 对**每一行表格数据**、**每一个关键事件**强制行级溯源。

**7.4.1 引用格式**

每一行数据的**末尾**（或紧跟的小字脚注）必须附三要素标签：

```
[来源: <MCP 工具名> · <变更记录编号 / 原始数据键> · snapshot=<YYYY-MM-DD>]
```

**实例**：

```markdown
| 2023-07-06 | 5,319.15 万元 | 36,000 万元 | +577% | 股改 + Pre-IPO 增资 |
  └ 来源: qcc-company/get_change_records · record_id=CCR-2023-07-06-001 · snapshot=2026-04-17

| 2019-10-14 | 万得信息 +18% 入场 | 战略投资者进入 | Wind 战略入股 |
  └ 来源: qcc-company/get_change_records · record_id=CCR-2019-10-14-003 · snapshot=2026-04-17
```

**7.4.2 三要素硬性要求**

| 要素 | 说明 | 缺失处理 |
|---|---|---|
| MCP 工具名 | 必须是实际调用过的工具（如 `qcc-company/get_change_records`） | 禁止出现在报告中 |
| 变更记录编号 / 原始数据键 | 工具返回中的 `record_id` / `business_id` / `case_no` 等唯一键 | 若原始返回无此键，使用 `date + field_name` 组合键 |
| snapshot 日期 | 本次会话实际调用 MCP 时的服务端数据快照日期（不是报告生成日） | 必填 |

**7.4.3 输出前校验脚本**（模板）

```python
# 伪代码：最终输出前 grep 检查
import re
PATTERN = r'\[来源: [\w\-/]+ · [^·]+ · snapshot=\d{4}-\d{2}-\d{2}\]'

def validate_provenance(report_md: str) -> list[str]:
    """
    对每一行表格数据（以 | 开头的行）检查是否有紧随其后的来源标签。
    返回缺失行列表；若非空，SKILL 必须停止输出并报错。
    """
    missing = []
    for i, line in enumerate(report_md.splitlines()):
        if line.startswith('|') and not line.startswith('|---'):
            # 下一行应包含来源标签
            next_line = report_md.splitlines()[i+1] if i+1 < len(report_md.splitlines()) else ''
            if not re.search(PATTERN, next_line) and '来源:' not in line:
                missing.append(f'Line {i}: {line[:80]}')
    return missing

# 在 SKILL 输出最终报告前调用：
# if validate_provenance(report) != []: raise ValueError(...)
```

**7.4.4 简化模式（便于可读性）**

为避免每行下方都堆一行脚注影响可读性，允许以下两种替代形式：

- **合并式**：表格增加"来源"列，填简短标识（如 `CCR-2023-07-06-001`），并在表下方统一给出 `snapshot` + 工具名。
- **章节末注**：在每个表格下方统一附 `> 数据来源：get_xxx（snapshot=2026-04-17）；record_id 详见表格"来源"列。`

但两种简化形式下，**三要素仍必须齐全**（工具名、记录编号、snapshot 日期），不得因简化而省略任一要素。

## 八、时间轴对撞逻辑详解

### 8.1 对撞矩阵

| 风险事件类型 | 对撞的工商变更类型 | 对撞窗口 | 洗白怀疑分 |
|---|---|---|---|
| 历史失信被执行 | 法代、住所、股东、对外投资 | ±3 月 | 高（×3） |
| 历史行政处罚 | 法代、经营范围、股东 | ±3 月 | 中（×2） |
| 历史被执行人 | 法代、住所、股东 | ±3 月 | 高（×3） |
| 历史经营异常 | 住所、法代 | ±6 月 | 中（×2） |
| 历史终本案件 | 法代、股东、对外投资 | ±3 月 | 中（×2） |

### 8.2 洗白怀疑分阈值（与 §6.1 评级标签联动）

- 累计分 0：整体评价标签 → **`clean_baseline`（一贯合规）**
- 累计分 1-5：视修复窗口是否伴随关键变更定档 → **`recovered`**（修复窗口无关键变更）或 **`evasive_high_risk`**（伴随）
- 累计分 ≥ 6：第七章"关键风险提示"必须列出，整体评价标签 → **`evasive_high_risk`**
- 累计分 ≥ 10：整体评价标签 → **`evasive_high_risk`**，并在风险提示首条加粗高亮

### 8.3 对撞呈现形式（在第三 / 四 / 五 / 六章内）

```markdown
| 2022-04-18 | 股东变更 | 深圳前海燧石鼎天 退出 -7.19% | 陈德强 +5.19% | — |
|            |          | ⚠️ 关联事件：2022-08-01 著作权纠纷（被告，赔偿 10 万元） · 时差 3.5 月 |
  └ 来源: qcc-company/get_change_records · record_id=CCR-2022-04-18-002 · snapshot=2026-04-17
  └ 对撞数据来源: qcc-history/get_historical_judicial_cases · case_no=XXXXX · snapshot=2026-04-17
```

> 对撞标注同样受 §7.4 行级溯源硬门控约束：每一条 ⚠️ 关联事件必须标注 **工商变更数据来源**与**对撞风险数据来源**两行。

### 9.4 示例 · 第四章 浓缩（治理结构变迁）

**4.1 法代变更**
| 变更日期 | 变更前 | 变更后 | 同日其他变更 |
|---|---|---|---|
| 2015-05-05 | 马群 | 陈德强 | 股东大洗牌 + 注册资本 +12.4% |

**4.2 大股东关键事件**
- 2015-05-05：陈德强 +64%，成为第一大股东
- 2019-10-14：**万得信息**（上市公司 Wind）+18%，战略投资者入场
- 2024-10-24：**万得信息 16.81% 由直接持股改为间接持股**，通过 上海荷花缘企业管理中心（有限合伙）持有；荷花缘合伙人为 万得信息 LP 99% + 上海万兴信息科技（万得 100% 全资子公司）GP 1%，**穿透 100% 仍归万得信息**，非真实退出

*专家点评（KYB 视角）：法代 10 年仅一次变更，控制权极稳。**2024-10 "Wind 退出" 经 MCP 穿透核实属误报**——承接方"上海荷花缘企业管理中心（有限合伙）"为 Wind 全资 SPV，实为持股架构由直接转为间接，最终受益人未变。KYB 受益人穿透时应将荷花缘视同 Wind 处理。*

*专家点评（投资尽调视角）：Wind 于 2019-10 入场持股 16.81%，2024-10 将直接持股调整为通过全资有限合伙间接持股，非实质退出。该动作动因可能是 Wind 自身资产重组、员工激励载体、或 Pre-IPO 结构预整理——建议在投资对话中向 Wind 直接求证此次架构调整的业务目的。*
