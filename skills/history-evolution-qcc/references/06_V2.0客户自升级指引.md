# V2.0 客户自升级指引

> V1.0 由 Anthropic + 企查查官方维护，**纯 MCP 驱动**，客户零材料负担。
> V2.0 由**客户自维护**，用于在 V1.0 之上追加自有尽调数据。本文档是指南，不是 V2.0 SKILL 本体。

## V1.0 / V2.0 分界线

| 判别标准 | 归 V1.0 | 归 V2.0 |
|---|---|---|
| 数据是否可从企查查 MCP 直接查询 | ✅ 是 | ❌ 否 |
| 是否依赖客户手工尽调 / 文档采集 | ❌ 不依赖 | ✅ 依赖 |
| 维护方 | Anthropic + 企查查 | 客户自有 |

## V2.0 典型补充字段

客户私域可补充以下 11 类（V1.0 明确剔除）：

1. **代持存在性声明与还原**（依据：律所核查意见 / 代持协议）
2. **一致行动关系声明**（依据：一致行动协议 / 招股书声明）
3. **独立性五维定性评估**（业务 / 人员 / 资产 / 机构 / 财务，依据：保荐机构工作底稿）
4. **股权激励细则**（行权价 / 授予条件 / 离职回购，依据：激励计划 + 董事会决议）
5. **关联交易明细**（对手方 / 金额 / 定价依据，依据：定期报告 + 关联交易公告）
6. **对赌 / 回购 / 业绩承诺 / 优先清算**（依据：投资协议 / SPA）
7. **同业竞争承诺 / 避免关联交易承诺**（依据：承诺函）
8. **股利分配政策 / 滚存利润安排**（依据：章程 + 股东会决议）
9. **上市锁定 / 减持 / 稳定股价 / 欺诈发行回购 / 填补即期回报**承诺体系（依据：招股书 + 承诺函）
10. **特别表决权 / AB 股 / VIE 架构**（依据：章程 + 境外协议）
11. **亲属关系定性网络图**（依据：尽调问卷 + 履历访谈）

### V1.1 追加（企查查 PPT 范式补充）

以下 2 类维度当前 MCP 无直接工具覆盖，归客户 V2.0 自补：

12. **行业解决方案叙事**（参照企查查 2025 公司介绍 PPT P14–P16）
    - **依据**：客户自有行业研究报告 + 方案 PPT + 客户案例（如金融行业尽调/风控方案 / 制造行业供应商管理方案 / 中小企业精准拓客方案 / 政府招商引资方案）
    - **内容建议**：按行业分组 4–6 个，每个行业含"核心痛点 + 解决方案流程图 + 典型客户"三段
    - **图表参考**：PPT P15 的"前期评估 → 中期审批 → 后期管理"三阶段流程图；P16 的"年均排查 5000+ / 节约 8000+ 小时 / 效率增长 2000%" 关键指标卡
    - **待 MCP 升级**：当企查查 MCP 未来上线「行业标签」「解决方案库」工具后，将自动进入 V1.x 正式章节

13. **合作客户 Logo 墙 / 供应商关系**（参照 PPT P23）
    - **依据**：客户关系管理系统（CRM）+ 采购管理系统（SRM）+ 合作协议
    - **内容建议**：按行业分组的大客户 Logo 网格（金融 / 制造 / 能源 / 互联网 / 政府），每行业 5–8 家代表客户
    - **注意事项**：Logo 使用需获得客户书面授权；避免暴露非公开客户关系
    - **待 MCP 升级**：**当企查查 MCP 未来上线「客户关系」「供应商关系」工具（例如基于招投标中标关系 / 发票关系 / 付款流水推断）**，将自动进入 V1.x 正式章节；此后客户 V2.0 补充逻辑可下线

**升级路径**：一旦这两类 MCP 工具上线，V1.x 的 build_docx.py 将直接渲染，无需客户再手工维护 Logo/叙事段落。客户 V2.0 段落可保留作为"自定义补充"放在报告附录。

## 推荐扩展方式

### 方式 A · 直接编辑 SKILL.md

**适用**：客户只需固定几家企业，批量生成。

1. Clone V1.0 到私域：`cp -r qcc-history-skill-v1.0 qcc-history-skill-v2.0-<org>`
2. 编辑 `SKILL.md`：在"条件扩展子节"表下追加客户自有子节
3. 编辑 `references/02_报告模板.md`：追加 V2.0 自定义章节模板
4. 编辑 `scripts/build_docx.py`：追加自有字段的渲染函数

### 方式 B · 外挂数据源 YAML

**适用**：客户有固定尽调底稿字段集合，需复用。

1. 在 V2.0 仓库根目录新建 `customer_data/` 目录
2. 每家企业一份 YAML：`customer_data/<企业名>.yaml`，格式示例：
   ```yaml
   company: 企查查科技股份有限公司
   credit_code: 913205066854783456
   v2_extensions:
     proxy_holding:
       - holder: 马群
         real_owner: 陈德强
         start: 2012-06
         end: 2015-05
         source: 招股书 P.241
     concerted_action:
       statement: 各股东间不存在一致行动
       source: 招股书 §4
     independence_assessment:
       business: 独立
       personnel: 独立
       assets: 独立
       organization: 独立
       finance: 独立
       source: 保荐机构工作底稿 W-2026-03-15
   ```
3. 修改 `build_docx.py` 让其在 MCP 数据之外，加载同名 YAML 的 `v2_extensions`
4. 每段 V2.0 内容自动加"客户自采"来源标签

### 方式 C · Word 表单 + 后端解析

**适用**：客户经理手工填写，不熟悉 YAML。

1. 用 Word 制作客户尽调表单（参考附件模板）
2. 客户经理填写，提交到共享目录
3. 用 `python-docx` 写解析脚本，把 Word 表单字段转成 YAML，再接入方式 B

## V2.0 段落溯源建议

在 V2.0 章节渲染中，每段内容追加来源标签：

```
[来源: 客户自采 · 尽调底稿 DD-2026-04-10-001 · 采集人：张经理]
```

与 V1.0 的 `[来源: qcc-xxx · snapshot=YYYY-MM-DD]` 明确区分，方便审计追溯与 V1.0 升级后自动替换。

## V2.0 不承诺兼容性

本 SKILL 作者对 V2.0 的修改**不负责兼容性**：

- V1.1 / V1.2 等后续版本如果把某字段纳入 MCP 覆盖（如 `qcc-history` 全量上线），V2.0 客户段落可能产生数据冲突
- V2.0 客户侧需自行处理合并 / 替换逻辑
- 建议 V2.0 段落与 V1.0 段落结构分离（独立章节块），便于未来迁移

## 版本命名建议

客户私域 V2.0 命名格式：

```
qcc-history-skill-v2.0-<组织缩写>-<发布日>
例：qcc-history-skill-v2.0-cmb-20260501（招商银行 2026-05-01 版）
```

## 常见 V2.0 错误模式

1. ❌ **在 V1.0 的 7 章里"暗改"字段**：会破坏 build_docx 的来源标签，导致审计失败
2. ❌ **把 V2.0 段落写回 MCP 字段映射表**：违反 V1.0 MCP 纯度原则
3. ❌ **用"推测" / "据客户透露"类软文字**：必须具体到文档编号 / 采集人 / 采集日期
4. ✅ **正确做法**：独立 §V2-X 章节 + 独立来源标签 + 独立扩展脚本

## 技术支持边界

- **V1.0 范围**：Anthropic + 企查查技术支持
- **V2.0 范围**：客户自维护。本 SKILL 作者不回答 V2.0 的字段扩展 / 脚本修改问题
- **例外**：如客户发现 V1.0 的 bug 或误判，仍走 V1.0 issue 通道反馈

## 未来迁移路径

- 若客户 V2.0 中的某字段经反馈被产品侧采纳，未来企查查 MCP 可能新增对应工具
- 工具上线后，下一个 V1.x 会自动覆盖该字段，客户 V2.0 对应段落应迁移或弃用
- 迁移过程：本 SKILL 作者通过 CHANGELOG 明确披露新增工具，客户对照迁移
